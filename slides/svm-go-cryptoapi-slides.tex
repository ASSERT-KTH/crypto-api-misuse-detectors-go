\begin{frame}[fragile]
    \frametitle{A Cryptographic API Misuse}
    {\large\textbf{Traefik, CVE-2025-66491.} ingress-nginx provider:}
    \vspace{0.5em}
        \begin{minipage}{0.78\textwidth}
        \begin{lstlisting}
InsecureSkipVerify: strings.ToLower(
    ptr.Deref(cfg.ProxySSLVerify, "off")) == "on",
        \end{lstlisting}
        \end{minipage}
    \vspace{0.2em}
    \normalsize
    \begin{itemize}
        \setlength{\itemsep}{0.5em}
        \item The user sets \texttt{proxy-ssl-verify: "on"}, expecting TLS verification to be \textbf{enabled}.
        \item Expression evaluates to \texttt{true}, \texttt{InsecureSkipVerify} becomes \texttt{true}, inverting security assumptions.
        %\item Disables SSL certificate verification between traefik instance and backend.
        \item TLS encryption remains, but backend authentication is disabled: Traefik may trust a spoofed backend.    
    \end{itemize}
\end{frame}

% NOTES
% Here is a concrete API misuse vulnerability: CVE-2025-66491 in Traefik, a Go reverse proxy and load balancer.
% The vulnerable code maps the annotation proxy-ssl-verify: "on" to InsecureSkipVerify = true, which reverses the intended meaning.
% So instead of enabling backend certificate verification, it silently disables it.
% TLS encryption still exists, but backend authentication is lost: Traefik may trust a spoofed or man-in-the-middle backend.
% Certificate-validation flaws have been exploited in the wild before, e.g., in a 2020 Windows CVE.

\begin{frame}
    \frametitle{Why Go}
    \begin{itemize}
        \setlength{\itemsep}{0.9em}
        \item \textbf{Go runs critical infrastructure:} Kubernetes, Terraform, Traefik, Let's Encrypt's Boulder.
        \item Cryptographic misuses reach production, for example CVE-2025-66491 in Traefik and CVE-2024-45337 in \texttt{golang.org/x/ssh}.
        \item Four Go detectors exist (\Gopher, \CodeQL, \Gosec, \SnykCode), but none have been systematically evaluated.

    \end{block}

\begin{frame}
    \frametitle{Research Questions}
    \begin{itemize}
        \setlength{\itemsep}{0.7em}
        \item \textbf{RQ1.} What is the design space of state-of-the-art tools for detecting cryptographic API misuses in Go?
        \begin{itemize}
            % \item[i.e.,] tool scope, supported rules, and detection approach.
            \item We survey the tools and consolidate a taxonomy of 14 Go crypto misuse classes.
        \end{itemize}
        \vspace{0.5em}
        \item \textbf{RQ2.} To what extent are crypto misuse detection tools for Go consistent with each other?
        \begin{itemize}
            % \item[i.e.,] Where do their detection results agree or disagree?
            \item We run all four tools on 328 security-relevant Go projects and compare their findings.
        \end{itemize}
    \end{itemize}

            \setlength{\itemsep}{0.35em}
            \item TLS stays encrypted, but peer authentication is gone.
            \item A machine-in-the-middle can present its own certificate and be trusted.
        \end{itemize}
        \vspace{0.15em}
        {\footnotesize\textbf{Overlap.} This is the largest class. All tools find true positives here, but \Gosec and \SnykCode contribute many tool-unique alerts, with different precision levels.}

    \textbf{2. Pattern match misses the equivalent} \hfill \textit{Rule \#13, only \CodeQL}
    \begin{verbatim}

    \end{verbatim}
    \CodeQL detects this pattern; the other tools do not report it in our study.

    \textbf{3. Field name flagged, value ignored} \hfill \textit{Rule \#11, \Gosec false positive}
    \begin{verbatim}
    tlsConfig := &tls.Config{Certificates: []tls.Certificate{cert}, RootCAs: roots} // flagged
	setTLSDefaults(tlsConfig) // MinVersion 1.2 = OK
    \end{verbatim}
    \Gosec lacks data flow capabilities and flags false positives.

\begin{frame}
    \frametitle{Implications}
    \textcolor{red}{WIP, add takeaway or something.}
    \begin{itemize}
        \setlength{\itemsep}{0.75em}
        \item \textbf{For practitioners:} no single detector is enough. The tools miss different things, so relying on one leaves systematic blind spots.
        \item \textbf{For evaluators:} raw counts mislead. On insecure PRNG detection, \Gosec reports 2,517 findings where \Gopher reports 0, on the same code and for the same rule.
        \item \textbf{For tool builders:} the gap is qualitative, not only quantitative. The tools differ in rule coverage and in how much program context they model.
    \end{itemize}
