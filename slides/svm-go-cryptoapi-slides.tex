\begin{frame}
    \frametitle{Problem Context}
    \begin{itemize}
        \setlength{\itemsep}{0.7em}
        \item Go is widely used in security-critical infrastructure, including certificate services, container platforms, and cloud tooling.
        \item Cryptographic API misuse remains a consequential vulnerability class: code can compile and pass tests while still violating core security assumptions.
        \item Unlike Java, the current landscape of cryptographic misuse detection for Go has not been systematically characterized.
    \end{itemize}
\end{frame}

\begin{frame}
    \frametitle{Evaluation Challenge}
    \begin{itemize}
        \setlength{\itemsep}{0.7em}
        \item Existing Go detectors do not operate over a common target: they differ in supported rules, reporting granularity, and execution robustness.
        \item Raw alert counts are therefore hard to compare: more findings may reflect broader rule coverage, noisier heuristics, or simply a different reporting unit.
        \item The key question is not only whether these tools detect crypto misuses, but also what they detect and how their outputs should be interpreted.
    \end{itemize}
\end{frame}

\begin{frame}
    \frametitle{Prior Work}
    \begin{itemize}
        \setlength{\itemsep}{0.6em}
        \item Comparative work already exists for Java crypto misuse detection, including multi-tool evaluation and false-positive analysis [1, 2, 3].
        \item Go-specific detector work also exists: \textit{CryptoGo} introduced the first Go detector, and \Gopher extended that line of work [4, 5].
        \item What is still missing is a systematic comparison of the Go detectors currently available in practice.
    \end{itemize}
    \vspace{0.5em}
    \begin{block}{This paper}
        We provide the first comparative study of currently available Go crypto misuse detectors.
    \end{block}
    \vfill
    {\tiny
    \begin{columns}[T,onlytextwidth]
        \column{0.48\textwidth}
        [1] Ami et al. ``Why Crypto-detectors Fail: A Systematic Evaluation of Cryptographic Misuse Detection Techniques.'' SP 2022.\\[0.2em]
        [2] Zhang et al. ``Automatic Detection of Java Cryptographic API Misuses: Are We There Yet?'' TSE 2022.\\[0.2em]
        [3] Chen et al. ``Towards Precise Reporting of Cryptographic Misuses.'' NDSS 2024.
        \column{0.48\textwidth}
        [4] Li et al. ``Cryptogo: Automatic Detection of Go Cryptographic API Misuses.'' ACSAC 2022.\\[0.2em]
        [5] Zhang et al. ``Gopher: High-Precision and Deep-Dive Detection of Cryptographic API Misuse in the Go Ecosystem.'' CCS 2024.
    \end{columns}}
\end{frame}

\begin{frame}
    \frametitle{Research Questions}
    \begin{itemize}
        \setlength{\itemsep}{0.7em}
        \item \textbf{RQ1.} What is the design space of current tools for detecting cryptographic API misuses in Go?
        \item \textbf{RQ2.} To what extent are crypto misuse detection tools for Go consistent with each other?
    \end{itemize}
    \vspace{0.5em}
    \begin{block}{Study Design}
        RQ1 compares tool scope, supported rules, and detection approach.

        RQ2 runs all tools on 328 security-relevant Go projects and compares coverage and overlap.
    \end{block}
    \vspace{0.4em}
    \begin{itemize}
        \setlength{\itemsep}{0.35em}
        \item Goal: establish what these tools detect, how they differ, and what follows for practice and evaluation.
    \end{itemize}
\end{frame}

\begin{frame}
    \frametitle{Manual Analysis}
    \begin{itemize}
        \setlength{\itemsep}{0.6em}
        \item For Rule \#05, overlap is not a correctness signal: 4 of 5 detections shared by \CodeQL, \Gosec, and \SnykCode but missed by \Gopher occur in examples or mock code.
        \item For Rule \#11, 3 of 5 sampled \Gosec-only detections are false positives; in our \SnykCode sample, we found no clear false positives, and the alerts better matched insecure TLS configurations.
        \item For Rule \#13, \CodeQL contributes unique SSH findings beyond \texttt{InsecureIgnoreHostKey} by reasoning about custom callbacks that always return \texttt{nil}.
    \end{itemize}
    \vspace{0.5em}
    \begin{block}{Interpretation}
        Cross-tool overlap is informative, but the central result is that the tools encode different notions of misuse and different amounts of program context.
    \end{block}
\end{frame}

\begin{frame}
    \frametitle{Implications for Practice and Evaluation}
    \begin{itemize}
        \setlength{\itemsep}{0.7em}
        \item \textbf{For practitioners:} using a single detector is likely to leave blind spots; using multiple tools is justified by the low overlap.
        \item \textbf{For evaluators:} raw finding counts are misleading when tools disagree this much within the same rule class.
        \item \textbf{For tool builders:} the differences are qualitative, not only quantitative: \Gopher covers the largest rule set, \SnykCode and \Gosec run most reliably, and \CodeQL contributes higher-overlap alerts plus distinct SSH findings.
    \end{itemize}
    \vspace{0.4em}
    \begin{block}{}
        Detector choice materially affects both deployment decisions and empirical conclusions.
    \end{block}
\end{frame}

\begin{frame}
    \frametitle{Takeaways}
    \begin{itemize}
        \setlength{\itemsep}{0.55em}
        \item Coverage, robustness, and agreement vary enough that detector outputs should be treated as complementary rather than interchangeable.
        \item For this problem, detector evaluation must compare what tools can detect, not only how much they report.
    \end{itemize}
\end{frame}

\begin{frame}
    \frametitle{Future Work}
    \begin{itemize}
        \setlength{\itemsep}{0.55em}
        \item Formalize and enforce API-level security invariants beyond today's rule-level checks.
        \item Extend the comparison as tools evolve and additional Go misuse classes become detectable.
    \end{itemize}
\end{frame}
